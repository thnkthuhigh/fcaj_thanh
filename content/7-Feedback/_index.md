After completing the FCAJ internship, I see it as a meaningful learning experience that stays quite close to a real working environment. What makes the program valuable to me is not only the number of workshops or the AWS content itself, but the way it pushes interns to connect cloud knowledge with a concrete project. That is the part that gave the internship real impact.

### What I think the program did well

The first strength is that the program creates a clear learning rhythm. Students do not stop at AWS theory; they are required to apply that knowledge to a product that has users, data, and technical problems. For me, that made backend learning through CloudDoc much more practical.

The second strength is the combination of workshops, technical materials, mentor feedback, and team project work. Backend is an area where it is easy to get stuck if you only study isolated tools. With mentor input and technology-sharing sessions, I better understood why a technical decision should be chosen, not only how to make it run.

The third strength is that CloudDoc is scoped well enough for learning while still being rich enough to develop system thinking. From the backend side, the project gave me the chance to work with API design, PostgreSQL, S3 upload flow, IAM-related security thinking, logging, and collaboration with the frontend. That kept the internship from feeling too narrow.

### What I learned from the overall experience

This program helped me realize that cloud is learned best in the context of a product. While working on CloudDoc, I could not think about backend only as “which endpoint returns which data.” I also had to think about metadata, storage, consistency, and operational concerns. That way of learning is harder, but it makes the knowledge stay with me longer and become more useful later.

I also saw more clearly the value of cross-role collaboration. A feature is only complete when frontend, backend, and infrastructure align well enough. Standalone labs rarely create that experience, but FCAJ did it quite naturally through the project process.

### What I hope the program can improve

The first thing I would improve is early technical guidance for project teams. If the program provided a concise guideline from the beginning on business flow description, API contracts, environment checklists, and responsibility boundaries between frontend, backend, and AWS, teams could align faster and spend less time finding direction on their own.

The second improvement would be more frequent short technical reviews across roles. From a backend point of view, many blockers are not large coding problems, but they still strongly affect integration. More regular checkpoints would help teams catch small misalignments earlier.

The third improvement would be clearer guidance on testing, monitoring, and cost control for cloud-based projects. This would help interns understand the difference between a working demo and a system designed with operational thinking in mind.

### Personal reflection

**Difficulty**

My biggest difficulty was learning across two layers at the same time: completing the backend work for the project while also expanding my understanding of AWS and real operational thinking. At some points, the amount of new information was large enough to feel overwhelming.

**How I handled it**

I handled this by staying close to the actual needs of the project. Whenever CloudDoc needed a specific backend capability, I focused on understanding that part first, read the relevant material, tried it in the codebase, and discussed it with the team when needed. That kept my learning grounded in real tasks instead of becoming too abstract.

**Future direction**

After the program, I want to keep growing in backend development with a stronger connection to cloud operations: logging, monitoring, access security, background processing, and cost optimization. I believe that is the necessary next step if I want to work on more realistic systems later on.

### Closing thoughts

If I were to recommend FCAJ to later students, I would still encourage them to join, especially if they want to learn cloud through product building rather than isolated service exercises. For me, the program did more than add knowledge. It changed the way I think about backend as a core layer that must support business flow, architecture quality, and operational stability at the same time.
