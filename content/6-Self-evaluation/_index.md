During my internship at **Amazon Web Services Viet Nam Company Limited** from **April 17, 2026** to **July 10, 2026**, I participated in the **CloudDoc** project with a primary focus on **backend development**. This experience helped me move from basic familiarity with APIs and AWS to a clearer understanding of how a backend should be structured for a product that has real data flow, business logic, and operational concerns.

### How do I see my role in this internship?

If I summarize my contribution, I see it as building the core layer behind the interface: organizing document data, implementing the required endpoints, connecting PostgreSQL, working with S3 through presigned URLs, and supporting stable integration for the frontend. The biggest thing I learned was not only how to write more APIs, but how backend design preserves consistency across the system.

Compared with the beginning of the internship, I improved the most in the way I think about systems. I no longer treat each feature as an isolated endpoint. Instead, I now think in full flow: where the request enters, what data must be validated, what belongs in the database, what should be stored in S3, and where I should observe the system when something fails.

### Three things I did well

First, I stayed focused on the backend core of CloudDoc instead of spreading my effort too widely. I prioritized work that directly affected the system, such as metadata structure, document APIs, upload flow, and data integration, so my contribution remained meaningful to the product.

Second, I showed strong self-learning ability. When I encountered topics such as presigned URLs, data lifecycle thinking, or modular API design, I was able to read documentation, compare it with the codebase, and apply it to real tasks instead of stopping at theory.

Third, I worked collaboratively. Backend cannot operate in isolation, so I had to coordinate closely with the frontend side to align input, output, and business states. That process helped me better understand the role of backend as part of a shared system rather than a standalone block of code.

### Areas I still need to improve

I still need to strengthen my operational mindset at a more production-like level. I now understand why logging, monitoring, alerting, and cost control matter, but I still need more hands-on experience to implement those parts confidently in a real environment.

Another area is explaining technical solutions more clearly and concisely. At times I understood the processing flow, but needed more time to communicate it in a way that others could immediately follow. For backend teamwork, that skill matters a lot.

Finally, I want more testing experience across a wider range of scenarios, especially failure cases, edge data, and situations involving synchronization between the database, storage, and APIs.

### Personal reflection

**Difficulty**

The biggest challenge for me was moving from writing isolated endpoints to designing a complete backend flow. In CloudDoc, I did not only need to return data to the frontend; I also had to understand why metadata must be structured, why upload should be delegated to S3, and why AWS permissions must be controlled carefully.

**How I handled it**

I dealt with that challenge by always returning to the business flow first. Instead of jumping directly into code, I broke each problem into smaller questions: what the user submits, what the backend must validate, what the database should keep, what S3 should keep, and what response the frontend should receive. That method helped me reduce confusion and stay aligned with the project goal.

**Future direction**

After this internship, I want to continue growing in practical backend engineering: writing stronger APIs, organizing data more effectively, testing more carefully, and understanding backend operation on AWS through logs, metrics, alerts, and cost optimization.

### Final self-assessment

If I assess myself honestly, I believe I completed my assigned work quite well and made a clear improvement in system thinking. The part I value most is that I no longer see backend as a place that only handles requests. I now see it as the layer that protects data consistency, business logic, and product stability when many components work together.
